## Installed app verification

- The supplied 9.408-second recording was extracted at approximately 56.5 frames per second. Frame inspection showed the language submenu disappearing as the pointer crossed between a right-edge parent menu and its left-opening child.
- An unchanged `v0.3.1` installation was sampled 80 times over 18.671 seconds, spanning an automatic data read. With the pointer stationary, neither menu layer changed; this ruled out the refresh loop as the trigger for the recorded path.
- The installed `v0.3.2` candidate showed exactly one More Actions menu containing the three language choices directly, with no child menu.
- The candidate was sampled continuously 180 times over 42.597 seconds while the menu remained open. The single menu and all language actions stayed present throughout a real automatic read; after the check, the panel showed a new successful update at `15:05:25` and the next-read countdown had restarted.
- Before a tag or release was created, the owner identified that a user unable to read Chinese could not know that language selection was hidden behind the localized More Actions label. Publication was paused and installed-app acceptance was reopened for a first-level Language entry.

## Code review

- Root cause: `v0.3.1` stabilized the AppKit status-bar menu, while the recording used a separate nested SwiftUI menu inside the floating panel. The previous endpoint-only check was applied to the wrong surface and could not detect intermediate disappearance.
- The final design uses two sibling, one-layer menus: Language exposes the three locale actions, while More Actions keeps the remaining tools. Forecasting, refresh cadence, reset-credit storage, and the AppKit status menu are unchanged.
- Competitor and OpenAI-window research was refreshed from current primary sources. The public positioning now treats Weekly Only as the current reliable product surface, not a promise that an upstream short window can never return.
