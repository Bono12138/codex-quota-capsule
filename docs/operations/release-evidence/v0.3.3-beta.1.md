# v0.3.3-beta.1 Release Evidence

Status: release candidate under verification
Updated: 2026-07-15

## Defect and root cause

- Public issue #23 records a P0 case where Codex's live weekly window advanced from 0% to positive usage while Quota Capsule kept displaying the first 0% reading and an obsolete reset time.
- Before first use, the upstream reset timestamp advanced with observation time for about 5.5 hours. The quality engine treated every shift beyond its five-minute jitter cluster as an unconfirmed reset candidate, but the sequence never reached the six-hour new-cycle threshold.
- Raw successful readings continued to enter SQLite. `QuotaRefreshReducer` correctly refused the unconfirmed candidate, which unintentionally froze the visible accepted snapshot instead of adopting the anchored first-use window.

## Automated verification

- The Swift regression first failed with `calibrating`, `shouldAdoptLiveSnapshot = false`, and a 0% forecast instead of the live positive reading.
- The shared cross-runtime fixture then failed in TypeScript with `calibrating` instead of `stable` before the equivalent implementation was added.
- The repaired rule follows a provisional reset only while every accepted active-cycle reading remains 0% and the reset shift matches the observation gap within five minutes. Positive usage ends that exception immediately.

## Installed app verification

- The installed branch candidate showed 5% used and a July 22 10:21 local reset while the live read-only probe returned the same 5% used, 95% remaining, and reset timestamp.
- The read-only SQLite query showed consecutive successful 4% and 5% raw readings for that same reset cluster.
- Across the next automatic refresh, the expanded panel stayed at the current 5% reading, preserved the 10:21 reset, and advanced its data-update timestamp instead of returning to calibration.
- The candidate was run from `/Applications/Quota Capsule Beta.app`; no Codex auth, configuration, binary, or login state was changed.

## Code review

- The exception is restricted to the active cycle, requires every accepted value in that cycle to remain exactly 0%, and requires reset drift to track the elapsed observation time within the pre-existing five-minute tolerance.
- Replacing earlier zero-only provisional observations avoids inventing a pace from a moving cycle start and bounds retained history without altering any confirmed positive-use segment.
- Genuine reset candidates, downward corrections, alternating streams, stale reads, and positive-use reset changes retain their prior quarantine behavior.
