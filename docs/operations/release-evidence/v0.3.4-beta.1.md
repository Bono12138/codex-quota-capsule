# v0.3.4-beta.1 Release Evidence

Status: released and verified from exact merged main
Updated: 2026-07-18

## Defect and root cause

- Public issue [#26](https://github.com/Bono12138/codex-quota-capsule/issues/26) records a P0 mismatch: Codex and a direct probe reported 2% used / 98% remaining while Quota Capsule remained at its previously accepted 0% reading.
- Successful 1% and 2% readings were already present in SQLite. The failure was therefore in the quality transition, not the Codex source adapter or percentage formatting.
- The 0.3.3 unused-window exception assumed uninterrupted polling and required provisional reset movement to match the observation gap. After an application gap, the reset shifted by about 4.5 hours: too different for that exception, but below the six-hour new-cycle threshold. The candidate was never added to a confirmation buffer and could calibrate forever.

## Repair

- A valid non-backward reset may replace zero-only provisional history even after sleep, restart, or a long polling gap. Positive use ends this exception immediately.
- Every persistent reset-time correction now has a confirmation exit path. Without a new-cycle signal, three consistent readings over two minutes rebase the same cycle instead of creating a false new cycle or remaining in calibration.
- Swift, TypeScript, and the shared cross-runtime fixture cover the application-gap sequence and the positive-use reset-time correction.
- Confirmation copy now says that the latest read succeeded while previous confirmed data is temporarily displayed.

## Code review

- The repair removes the unreachable middle range in the reset state machine: a stable reset correction no longer needs to be either five-minute jitter or a six-hour new-cycle jump.
- The zero-only rebase cannot rewrite positive-use evidence because it is disabled as soon as any accepted active-cycle reading is above 0%.
- A confirmed correction without a new-cycle signal keeps the same cycle identifier and rewrites its canonical reset; a confirmed six-hour shift or two-point usage drop retains the existing new-cycle behavior.
- A pre-merge diff review checked the zero-only boundary, same-cycle canonical-reset rewrite, retained new-cycle thresholds, Swift/TypeScript parity, privacy boundary, and release metadata; no unresolved critical or important issue remained.
