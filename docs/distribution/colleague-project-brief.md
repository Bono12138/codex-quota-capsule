# 给同事介绍 Quota Capsule

日期：2026-07-14
用途：可直接转发或作为口头介绍提纲

## 30 秒版本

Quota Capsule 是一个面向 Codex 重度用户的 macOS 本地额度续航工具。

现在市面上已经有 CodexBar、ClaudeBar、codexU、`usage`、onWatch、Quota Float 等额度监控产品，它们在多 provider、历史报表、账号管理或跨平台上各有优势。Quota Capsule 不准备再做一个大而全 dashboard，而是专门回答一个工作问题：**照现在这个速度，我的周额度能不能撑到重置；为了稳妥，未来 24 小时最多还能用多少。**

它常驻桌面和菜单栏，从第一个有效读数就给初步范围；历史变多后再提高置信度。数据读取和计算默认都在本机，不切换账号，不修改 Codex 配置，也不会自动使用重置券。

项目：<https://github.com/Bono12138/codex-quota-capsule>

## 为什么现在值得做

OpenAI 当前账户的额度界面和 app-server 可能只返回周窗口，而且公开说明并不保证窗口结构长期不变。一个简单百分比无法回答：

- 27% 已用到底算快还是慢？
- 当前速度会不会在重置前见底？
- 今天还可以放心用多少？
- 预测为什么会变化，当前数据是不是过期？

Quota Capsule 的产品目标不是制造更多额度信息，而是减少用户反复检查和不必要的额度焦虑。

## 相比现有产品，我们的优势是什么

### 1. 以“能不能撑住”作为第一任务

很多工具首先展示多个 provider、多个窗口、token、成本或账号。Quota Capsule 第一屏先给人话判断，再允许用户展开查看证据。

### 2. 不假装预测很精确

上游额度是整数百分比，天然有量化误差。产品显示预测范围、置信原因和证据冲突，而不是给一个看似精确的单点答案。

### 3. 从第一天就有价值

第一个有效周额度读数就会产生低置信初步判断；不会让用户等固定 6 小时。后续融合当前周期、近期活动和历史，再逐步收窄范围。

### 4. Weekly Only，避免当前无效信息

当前数据只返回周窗口时，正常界面不显示不存在的额度占位，也不按 `primary` / `secondary` 的位置猜窗口类型。若未来出现新的稳定窗口，再单独评审是否加入。

### 5. 本地、只读、边界窄

它不做多账号切换、不写 Codex auth/config、不管理 session、不自动兑换重置券。prompt、代码、session 正文和凭据不会进入产品历史。

### 6. 重置券到期和历史被认真对待

每张券在详情底部显示到分钟的本地到期时间，并保存隐私安全的生命周期历史，便于以后研究发放规律；“有券”不会被算法误当成当前周额度安全。

## 与几类竞品的关系

| 如果同事问 | 可以这样回答 |
| --- | --- |
| 为什么不用 CodexBar / ClaudeBar？ | 它们适合多 provider 监控，成熟度和分发更强；Quota Capsule 更专注 Codex 周额度续航判断。 |
| 为什么不用 onWatch？ | onWatch 的历史和 burn-rate 分析很强，适合 dashboard、团队或 FinOps；Quota Capsule 追求原生、低打扰、扫一眼判断。 |
| 为什么不用 `usage` / codexU？ | 它们在 token、成本、项目趋势和完整工作看板上更强；Quota Capsule 刻意少展示信息，把预测区间和 24 小时预算放在第一层。 |
| Quota Float 也会悬浮，有什么不同？ | 它是最接近的形态竞品；Quota Capsule 的重点是公开可审计的周预测证据、置信度、24 小时预算和发布回归约束。 |
| 为什么不做账号切换？ | 账号与配置写入会显著扩大信任和故障边界；当前 MVP 只解决额度判断。 |

## 我们不应该夸大的地方

- 目前只支持 macOS 和 Codex，provider 广度不如成熟竞品。
- 仍是未公证 Beta，安装体验不如 Homebrew/签名应用成熟。
- 历史 token、成本和项目分析不如 `usage`、codexU、onWatch 丰富。
- 预测受上游整数百分比和个人历史长度限制；“诚实显示不确定性”不等于永远准确。
- 项目仍需要更多真实用户数据验证状态阈值和文案是否好懂。

## 2 分钟口头介绍

我做 Quota Capsule 的起点，是发现“还剩多少百分比”并不能帮助我安排工作。比如周额度已经用了 30%，如果本周才开始一天和已经过去五天，风险完全不同；如果刚跑完一个特别重的任务，短期速度也不应该被直接外推一整周。

我调研了目前活跃的工具。CodexBar 和 ClaudeBar 很适合多 provider 菜单栏监控；`usage`、codexU 和 onWatch 有更丰富的 token、成本、历史和 burn-rate 分析；Quota Float 也验证了悬浮小组件的需求；Codex Quota Viewer 和 Quotio 则进一步做了账号、配置或路由管理。这说明额度可见性是真需求，但也说明再做一个功能表更长的 dashboard 没有意义。

Quota Capsule 的取舍是把产品收窄成“周额度续航判断”。它读取本机 Codex 的真实周窗口，把时间进度、额度变化、当前活动和历史证据结合起来，给出能否撑到重置、重置时可能剩多少，以及未来 24 小时建议最多用多少。因为上游只给整数百分比，产品会显示范围和置信原因，不会假装知道一个非常精确的答案。

它默认本地、只读，不切账号、不改配置、不自动使用重置券。桌面胶囊负责低打扰判断，详情面板负责解释。当前 OpenAI 对额度结构正在变化，所以产品先做 Weekly Only；但底层仍按实际窗口时长识别，未来可以适应上游再变化。

如果这个方向成立，它的价值不是“替用户看一个数字”，而是把 quota 变成可以采取行动的预算建议。
