# Codex rate-limit 实时更新可行性记录

日期：2026-07-01

## 结论

当前版本不把主胶囊设计成“实时推送更新”。

继续采用：

- 启动时立即读取一次。
- 后台每 60 秒轮询。
- 菜单栏保留手动刷新。
- 读取失败时显示失败原因，并保留上次成功数据。

主胶囊不再展示看起来像可点击按钮的刷新图标。刷新动作放在菜单栏和后续失败态 retry 里。

## 原因

`codex app-server` 的 JSON-RPC 流里可能出现非请求响应类通知，例如：

```text
account/rateLimits/changed
```

但把它作为默认实时更新机制还缺几个产品级前提：

- 需要长时间保持 app-server 进程和管道连接。
- 需要处理 app-server 退出、超时、Codex 同步、睡眠唤醒、网络变化后的重连。
- 需要验证这个事件是否覆盖所有 quota window 变化，而不是只覆盖某些 app-server 内部状态变化。
- 需要验证没有事件时是否仍会定期补读。
- 需要确认长期监听不会增加 Codex CLI 资源占用或造成误判。

因此，在没有完整稳定性验证前，不能把“实时更新”作为用户承诺。

## 当前产品处理

当前用户看到的是低打扰的近实时状态，而不是严格实时状态。

文案和 UI 应避免暗示：

- 一定实时。
- 每次 quota 变化都会立即更新。
- 用户必须点主胶囊上的刷新按钮。

可以说明：

- 每 60 秒自动刷新。
- 菜单栏可手动刷新。
- 失败时会保留上次成功数据并显示最近失败。

## 后续验证路径

如果后续要做实时更新，需要单独开任务：

1. 做一个长连接 app-server listener。
2. 记录 30-60 分钟内 notifications、read responses、transport exits。
3. 验证睡眠唤醒和 Codex CLI 同步期间的行为。
4. 事件触发后主动调用 `account/rateLimits/read`，不要直接信任 notification payload。
5. 保留 60 秒或更低频率轮询作为兜底。
6. UI 只在验证通过后再承诺“实时”。
