# 重置券使用时机优化：数学模型评审与产品化边界

日期：2026-07-14

状态：研究结论；不属于 `v0.3.1-beta.1` 的自动建议功能

## 结论

动态规划方向是正确的，而且能成为 Quota Capsule 后续很有价值的独立能力。但现有推导还不能直接给用户“最佳兑换时间”，也不应该把 `15.4%/天` 一类阈值写进当前产品。

这次版本先完成两项前置基础：

1. 修正周速度估计，避免把轮询量化误差当成真实不确定性；
2. 长期保存每张重置券的真实发放、到期和消失状态，为规则验证和回测积累数据。

之后再实现独立的重置策略求解器。

## 现有推导中正确的部分

### 1. 使用重置券会改变状态，不是单纯增加额度

应把一次完整重置建模为状态转移：

\[
q(s^+)=Q,\qquad T_{next}=s+L
\]

其中：

- \(q\) 是剩余额度；
- \(Q\) 是完整额度；
- \(T_{next}\) 是下一次自然重置时间；
- \(L\) 是新周期长度；
- \(s\) 是兑换时刻。

这个转移会同时产生三种效果：补满额度、丢弃兑换前余额、移动后续自然重置相位。只比较“这张券补回多少”一定会算错。

### 2. 多张券必须联合考虑

第一张券的兑换时间会改变下一次自然重置，也会改变后续券到期前可用的时间，因此不能逐张独立贪心。

### 3. 动态规划适合求确定性全局最优

当未来每个时段的需求和价值都已知时，Bellman 递推可以枚举等待、兑换和自然重置，求出给定时间网格内的全局最优策略。

### 4. 相同效果的券存在最早到期优先支配关系

如果两张券的重置范围、效果和限制完全相同，较晚到期券在较早到期券仍可使用时被兑换，可以交换两张券的身份而不改变额度路径，并留下更晚到期的券。因此至少存在一个按到期顺序使用的最优解。

这个结论不适用于重置类型、适用窗口或效果不同的券。

## 不能直接产品化的部分

### 1. `15.4%/天` 不是系统常数

该阈值来自特定的：

- 当前余额；
- 三张券到期日；
- 恒定日均需求；
- 人为选择的规划终点；
- 终点价值设定。

改变规划终点 \(H\) 就会改变阈值，因此它不能成为通用产品规则。

### 2. 恒定日均需求与真实使用不符

Codex 消耗通常呈现任务突发、长任务、模型切换和空闲期。把最近数小时线性外推为未来数周，正是当前周预测出现误导的风险之一。

求解器必须接受分时需求或需求情景，而不是只接受一个日均数。

### 3. 最大化完成额度不等于最大化用户价值

不同任务的重要性不同。用 10% 额度完成关键发布，可能比用 30% 额度完成低价值探索更重要。目标函数需要价值权重：

\[
\max_\pi E\left[\sum_t v_t y_t\right]
-\lambda_c E[N_{credits}]
-\lambda_w E[W_{discarded}]
-\lambda_r Risk(\pi)
+\Phi(S_H)
\]

其中：

- \(v_t\) 是任务价值；
- \(y_t\) 是被满足的需求；
- \(N_{credits}\) 是使用券数；
- \(W_{discarded}\) 是重置时丢弃的余额；
- \(Risk\) 是额度中断风险；
- \(\Phi(S_H)\) 是终点剩余额度、剩余券和重置相位的价值。

没有终点价值时，求解器会在规划边界附近产生人为策略。

### 4. 确定性计划不是在线最优策略

用户每天会得到新的额度读数，也会改变工作计划。真正需要的是反馈策略：在每次读数后重新估计未来需求并决定等待或兑换，而不是今天固定未来三张券的时间。

### 5. 产品机制仍需观测验证

用户实测表明完整重置会把下一次周重置移动到兑换时刻约七天后，但当前公开帮助文档没有把这个状态转移写成稳定契约。求解器必须把 `resetEffect` 和 `cycleLength` 作为参数，并在本地通过兑换前后快照验证，不能硬编码成永恒规则。

## 推荐的随机动态规划模型

### 状态

在离散时刻 \(t\) 定义：

\[
S_t=(q_t,T_t,C_t,z_t)
\]

- \(q_t\)：当前剩余额度；
- \(T_t\)：下一次自然重置；
- \(C_t\)：尚可使用的券，包含类型、到期、效果；
- \(z_t\)：使用强度状态，例如空闲、普通、突发，以及其后验概率。

### 动作

\[
a_t\in\{wait\}\cup\{redeem(i):i\in C_t\}
\]

首个开源版本只给建议，不自动兑换。

### 随机需求

\[
D_t\sim P(D_t\mid z_t,\text{近期历史},\text{用户计划})
\]

需求模型至少组合：

- 当前周期累计速度；
- 最近真实观察区间；
- 活跃/空闲分段；
- 历史相似周期；
- 用户手工标记的未来高强度时段。

### 状态转移

等待时：

\[
y_t=\min(q_t,D_t),\qquad q_{t+1}=q_t-y_t
\]

到达自然重置时：

\[
q_{t+1}=Q,\qquad T_{t+1}=T_t+L
\]

兑换第 \(i\) 张券时：

\[
q_t^+=Q_i,\qquad T_t^+=f_i(t,T_t)
\]

函数 \(f_i\) 由该券的 `resetType` 和经本地验证的规则决定。

### 风险策略

面向普通用户可提供三种参数化目标：

- 节省券：提高 \(\lambda_c\)；
- 平衡：最小化期望未满足需求和券浪费；
- 保守：最小化高分位或 CVaR 额度中断风险。

UI 必须同时展示建议的关键假设和对需求情景的敏感性，不能只给一个看似精确的时间点。

## 与当前周预测算法的接口

当前 `WeeklyRunwayPredictor` 应只回答：

- 本周当前节奏是否可能撑到重置；
- 未来 24 小时建议使用多少；
- 观察依据和不确定性是什么。

未来的 `ResetCreditOptimizer` 再消费：

- 周预测生成的多个需求情景，而不是一个点估计；
- 当前余额和下一次自然重置；
- 每张券的真实发放、到期、类型和状态；
- 用户选择的风险偏好和未来任务计划。

两层必须分开。优化器不能反过来修改原始周速度证据，也不能因为“有券”就把偏快状态显示为安全。

## 数据路线

### 阶段 1：本地事实记录

本次版本完成：

- 每张券的 `grantedAt`、`expiresAt`、类型和状态；
- 首次/最后看到时间；
- 到期、疑似兑换和未知消失分类；
- 兑换前后的周窗口变化。

### 阶段 2：确定性离线求解器

建立纯函数求解器和 JSON 参数格式，用人工需求计划验证 Bellman 递推、事件顺序、券到期和周期移动。

### 阶段 3：随机反馈策略

使用多种需求情景、风险偏好和滚动重算，比较：

- 立即兑换；
- 接近耗尽兑换；
- 到期前兑换；
- 动态规划反馈策略。

### 阶段 4：回测后进入 UI

只有当历史回测证明建议优于简单基线，且能稳定解释主要影响因素时，才在产品底部增加“重置建议”。

## 可配置参数

公开求解器至少需要：

| 类别 | 参数 |
|---|---|
| 当前状态 | 当前时间、余额、下一次自然重置或其区间 |
| 周期规则 | 完整额度、周期长度、自然重置和兑换的事件顺序 |
| 重置券 | 数量、类型、发放时间、到期时间、重置后额度、周期移动规则 |
| 需求 | 分时需求情景、情景概率、活跃状态转移 |
| 任务 | 时段价值、不可延迟任务、可延迟任务 |
| 风险 | 券机会成本、余额浪费成本、中断惩罚、CVaR 水平 |
| 操作 | 兑换延迟、安全缓冲、允许操作的时间网格 |
| 规划 | 滚动规划长度、终点价值、重算频率 |

## 资料可信度

- OpenAI 帮助中心和 `openai/codex` app-server schema 是机制与字段的主要来源。
- 本机接口返回的账户数据是个人券发放时间的主要来源。
- 社交媒体只能验证公开发布、全局重置或活动背景；无法证明某张个人券的具体来源。
- 目前公开社交媒体检索没有找到能够可靠解释每一个本机发放时间的官方事件，因此不对个人券来源作推断。
