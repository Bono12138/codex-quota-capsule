# Bug Triage 与发布阻塞规则

日期：2026-07-01
最近更新：2026-07-14

## 基本原则

Bug 的唯一执行来源是 GitHub Issues。

这个文档不维护完整 bug 列表，只定义：

- 什么算 P0 / P1 / P2。
- 什么阻塞内测。
- 什么阻塞正式发布。
- issue 应该怎么写。
- 修复完成前必须验证什么。

这样可以避免 Markdown 清单和 GitHub issue 双重维护导致状态过期。

## 优先级定义

### P0: 必须先修

满足任一条件就是 P0：

- 误导用户额度状态。
- 第一次有效读数仍不给任何周续航价值，或低证据估算被包装成确定结论。
- 预算公式暗中扣减未向用户解释的余量，或 Swift/TypeScript 对同一场景给出不同状态。
- 读取失败被显示成“够用”。
- 额度用完却显示“数据暂不可用”。
- 找不到 Codex CLI 但没有可操作诊断。
- app 无法启动、无法退出、频繁卡死。
- 会读取、记录或上传不该碰的敏感数据。
- 会修改 Codex auth、配置或登录状态。
- 破坏主体验：胶囊不可用、菜单栏状态明显错误、刷新逻辑失效、菜单栏不可见且胶囊展开面板没有反馈或设置兜底入口。

P0 必须在新功能前处理。

### P1: 本版应处理

满足任一条件通常是 P1：

- 内测用户会高频遇到，但不直接误导额度判断。
- 错误文案不清楚，导致用户不知道下一步怎么修。
- 三语基础文案缺失或严重不自然。
- 反馈入口缺失，影响内测问题收集。
- 主胶囊交互有明显歧义，例如刷新图标像按钮但不能点击。
- 停靠隐藏态残留详情面板、空白底板或大块透明内容。
- 发布文档缺关键路径，导致用户无法按说明安装。

### P2: 可排期

满足任一条件通常是 P2：

- 不影响当前主体验。
- 需要更多产品设计或用户反馈。
- 需要引入较多新状态、新页面或新持久化。
- 主要服务正式发布后的增长或运营。

例子：

- 历史趋势页。
- 通知。
- 自动更新。
- 主题。
- CSV / JSON 导出。
- Chrome 独立版真实 source。

### P3: 暂缓或实验

满足任一条件通常是 P3：

- 与主营体验距离较远。
- 可能增加隐私解释成本。
- 可能把产品推向 dashboard 或复杂监控工具。
- 需要大量平台特定工程。

例子：

- 语音和音效。
- 遥测。
- 多 provider dashboard。
- 敏感高级工具箱：账号、session、config。
- Windows native。

## 内测阻塞项

以下问题存在时不能发给朋友内测：

- Codex CLI 找不到时只显示“数据暂不可用”，没有说明原因和修复方向。
- 真实 quota 能读取到 100% 已用，但 UI 没有显示“已用尽”。
- 读取失败后清空上次成功状态。
- Codex app-server 瞬时失败没有重试，导致界面频繁显示读取失败。
- 启动首轮读取阶段没有显示“读取中”。
- 菜单栏和浮窗显示不同状态。
- 菜单栏状态项缺失时，桌面胶囊展开面板仍必须能直接刷新、提交反馈、通过带地球图标和 `Language` 文本的同级入口切换语言，并打开“更多操作”；新手引导、作者信息、关于页和退出从“更多操作”继续进入，高级数据设置通过“关于与反馈”继续进入。
- 点击折叠胶囊无法展开，或点击后被误判为拖动。
- 展开面板按钮被胶囊背景点击逻辑截获，导致 GitHub、邮件、作者 X、设置或关于入口无法触发。
- 展开面板按钮点击后被误判为胶囊背景点击，导致面板折叠。
- 胶囊只能在收起态调整宽度，展开态左右边缘无法调整宽度。
- 胶囊 resize 热区贯穿整个展开面板，导致详情区边缘按钮或胶囊顶部单击被吞，表现为无法稳定展开或收起。
- 靠边后没有进入完整可见的停靠迷你形态。
- 停靠隐藏态下面还显示展开详情或大块内容。
- app 无法退出。
- UI 文字明显截断，导致来源、接口、错误原因不可读。
- 周额度重置后，weekly window 已返回 100% 余量但“本周压力”显示成“暂时没有周窗口数据”。
- 周窗口刚开始时，少量整数百分比变化被线性外推成“本周额度很快耗光”。
- 周窗口刚开始时只显示等待/校准而没有初步估算和低置信说明。
- 周额度重置时间和数据更新时间混用；用户看不到下次自动读取倒计时。
- 突发使用后长时间空闲，活动速度仍保持峰值且不衰减。
- 下降修正或历史碎片被当成新消耗/完整历史先验。
- 周额度 reset 抖动或单次用量回退被误判成新周期，导致速度、重置时间或风险状态被误导。
- 尚未使用的周窗口在 0% 阶段返回滑动的临时重置时间，第一次正向用量出现后界面仍冻结在 0%、“正在校准”或旧重置时间。
- 休眠、重启或长轮询间隔后，稳定的实时读数与官方周额度一致，但胶囊因小于六小时的重置时间修正而无限校准并继续显示旧数据。
- 平坦轮询次数越多，activity 测量误差和重置余量范围反而越宽；同一真实变化必须不受轮询密度影响。
- 跨零预测被截成 `0%–Y%`，让用户误以为 0 是精确下界；主界面用难懂的 `%/天` 代替真实观察窗口。
- 状态栏菜单因每秒时钟或后台刷新而重建，导致语言子菜单闪烁、关闭或无法稳定选择。
- 悬浮面板使用嵌套或二级悬停语言菜单，在靠近屏幕边缘时因子菜单反向弹出和鼠标跨层路径产生闪烁或难以选择。
- 语言切换藏在随当前 locale 翻译的“更多操作”里，导致看不懂当前语言的用户无法发现恢复入口。
- 简中、繁中、英文基础界面缺失。
- 任一语言出现点击、缩放、展开、收起或文字截断问题，但没有横向检查另外两种语言。

## 正式发布阻塞项

以下问题存在时不能公开发布：

- 内测阻塞项未解决。
- README / INSTALL 不能让 Codex 用户按步骤完成安装。
- 隐私边界没有写清楚。
- public 仓库包含内部交接、私人上下文或未确认策略。
- 反馈入口缺失。
- public issues 模板没有提醒用户不要粘贴 token、cookie、auth 文件或 session 正文。
- 未说明当前是否签名、公证、是否需要右键打开。

## Issue 写法

Bug issue 必须包含：

- 发生了什么。
- 期望行为。
- 当前显示状态。
- 是否能在终端运行 `codex --version`。
- 是否能运行 `npm run probe:codex:rate-limits`。
- app 错误文案或截图。
- 不含敏感信息的日志。

Feature issue 必须包含：

- 用户问题。
- 为什么现在需要。
- 是否进入默认层、解释层、历史层、设置层、高级层或敏感层。
- 是否影响主胶囊 less is more 原则。
- 是否涉及隐私、上传、通知或账号状态。

## 修复完成标准

P0 / P1 bug 关闭前至少需要：

- 对应自动化测试，或说明为什么无法自动化。
- 本地跑过相关验证命令。
- 如果涉及 UI，提供截图、录屏或人工验收说明。
- 如果涉及 Codex source，确认没有修改 Codex auth、配置或登录状态。
- 如果涉及文案，确认简中、繁中、英文三语同步。
- 如果涉及预测，必须有 Swift、TypeScript 或共享 fixture 的回归测试，并同步更新预测方法和 changelog。
- 如果涉及 polling、预测融合或菜单生命周期，必须分别覆盖轮询等价 fixture、正负/跨零投影以及菜单跟踪期间的去重和延迟应用测试。
- 如果涉及真实读取或时间显示，必须核对已安装 app 的界面、数据库、进程路径和至少一次真实读数/空闲变化；不能只凭 mock 宣布修复。
- 如果 bug 先在某一种语言下出现，修复时必须判断它是语言文案问题、布局容量问题还是通用交互问题，并把结论同步到另外两种语言。
