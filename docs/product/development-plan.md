# 开发规划与需求采纳记录

日期：2026-07-01

## 规划原则

Quota Capsule 的默认体验只回答一个问题：

```text
按当前速度，用户还能不能撑到下一次刷新？
```

任何新功能都要先判断它属于哪一层：

1. 默认层：主胶囊扫一眼能懂。
2. 解释层：点击后说明为什么。
3. 历史层：用量快照和趋势。
4. 设置层：显示、通知、主题、provider。
5. 高级层：CLI、JSON、导出、多 provider。
6. 敏感层：账号、session、config，默认不做。

默认层必须保持克制。不能为了功能丰富把主胶囊做成 dashboard。

## 本轮开发目标

本轮目标是把产品从“自己能跑”推进到“可以稳定给少量用户内测”。

优先级：

1. P0 稳定性和诊断。
2. 三语界面和首屏语言策略。
3. 本地历史数据、远程产品分析和用户使用数据采集。
4. 反馈和关于入口。
5. 发布渠道与 public 仓库准备。
6. 刷新机制和刷新按钮设计确认。

## 下一版完整开发范围

目标：把当前版本从“功能可用”推进到“新用户能理解、老用户愿意常驻、后续分析有数据基础”。

下一版按六条线并行推进：

1. 新手引导和交互链路。
2. 胶囊与详情面板的信息架构。
3. 本地历史数据。
4. 远程产品分析和用户使用数据采集。
5. 三语界面和首屏语言策略。
6. 视觉质量、二维码可用性和完整验证。

### A. 新手引导返工

状态：采纳，下一版 P1。

问题：

- 当前引导只有一个说明页，不能带用户理解每个区域的作用。
- 文案避免先提出一个用户没有想到的类别，再用否定句解释产品定位。
- 不能默认简中首屏。系统语言命中简中、繁中、英文时直接使用对应语言；否则显示语言选择。
- 隐私说明不能藏在一个只展开一两句话的按钮里，核心隐私边界直接展示。

需求：

- 首次启动自动展开主胶囊和详情面板。
- 分步介绍：常驻胶囊、详情面板、周预测、菜单栏、反馈入口。
- 用户每一步都能看到对应区域被高亮。
- 支持跳过和重新打开引导。
- 读取失败时进入诊断引导，展示可执行下一步。

### B. 胶囊、面板和菜单栏重排

状态：采纳，下一版 P1。

需求：

- 菜单栏保持最短信息：`安全 17%`。
- 未展开胶囊要比菜单栏多一点价值：状态、5 小时已用百分比、微型时间/用量对比，空间足够时显示预计剩余。
- 胶囊支持有限尺寸调整，持久化宽度；避免增加一堆显示模式。
- 展开面板重新排版，消除重复信息。
- `刷新时预计还剩 76%` 只出现一次，不再和“刷新余量 76%”进度条重复。
- 当前速度只显示倍率和短解释，不显示进度条。
- 周预测作为低注意力区域展示。

### C. 本地历史数据

状态：采纳，下一版 P1。

原则：

- 本地历史数据服务于用户自己的趋势回看、诊断和后续导出。
- 不记录 prompt、session、token、cookie、API key、文件路径、项目名或可反推具体工作内容的数据。
- 远程产品分析由 D 节单独管理，不能和本地历史库混写成一个含糊功能。

实现范围：

- 在 macOS app 中加入 SQLite 本地历史库。
- 每分钟记录 quota snapshot；5 小时窗口和 weekly 窗口分别记录。
- 记录读取失败 snapshot，但做节流，避免失败时高频刷写。
- 加入本地事件指标，例如 app 启动、胶囊展开、胶囊尺寸调整、手动刷新、读取失败、反馈入口点击、引导完成。
- 提供本地数据清空能力。
- 后续本机分析页面从 SQLite 读取。

### D. 远程产品分析和用户使用数据采集

状态：采纳，下一版 P1。

原因：

- owner 明确需要看到真实用户如何使用产品，便于迭代产品、判断常驻价值和发现卡点。
- 只做本地历史库无法回答“多少用户每天打开”“常驻胶囊用了多久”“新手引导卡在哪一步”“哪个版本读取失败最多”这些产品问题。
- 远程采集必须有清晰边界，避免用户误解为读取工作内容或 Codex 会话内容。

原则：

- 首次引导里用当前界面语言说明采集范围，并提供开启 / 关闭选择。
- 设置里长期提供开关、采集项说明和本地清空入口。
- 只上传产品使用事件、粗粒度环境信息和错误类型。
- 不上传 prompt、session、token、cookie、API key、原始 Codex session 文件、文件路径、项目名、窗口标题、终端命令、代码内容或任何可反推具体工作内容的数据。
- 使用随机安装 ID 或匿名设备 ID，不收集真实姓名、邮箱、社交账号。

下一版采集事件：

- app 启动、退出、运行时长。
- 悬浮胶囊显示时长、隐藏时长、展开 / 收起次数、尺寸调整。
- 菜单栏常驻开启状态。
- 手动刷新次数、自动刷新成功 / 失败、失败类型。
- 当前状态分布：安全 / 注意 / 危险 / 未知。
- 首次引导开始、每一步到达、完成、跳过、失败诊断触发。
- 反馈入口点击：Email、GitHub Issues、X、抖音打开、抖音号复制。
- 版本、locale、macOS 主版本、CPU 架构、显示语言。

下一版工程范围：

- 建立远程 analytics 事件字典和 schema version。
- 增加用户同意状态、开关和采集说明。
- 增加本地事件队列，网络失败时可重试并限制队列大小。
- 增加上报 adapter；endpoint 未配置时只记录队列和开发日志，不静默发送。
- 在开发文档里记录后端候选方案和发布前必须确认的 endpoint。

### E. 三语界面和首屏语言策略

状态：采纳，下一版 P1。

原则：

- 简中、繁中、英文从第一版公开内测开始就是正式需求，不能作为后续补丁。
- 首次打开时先判断系统语言。命中 `zh-Hans`、`zh-Hant`、`en` 时直接显示对应语言。
- 系统语言无法匹配时，先显示语言选择，再进入引导。
- 引导、菜单栏、详情面板、反馈、关于、诊断、设置、隐私和 analytics 同意说明都必须覆盖三语。
- 繁中文案按台湾、香港、新加坡用户可自然理解的表达审校，避免简中机械替换。

下一版工程范围：

- 建立完整 string catalog 或统一文案表。
- 为三语补齐 onboarding、analytics consent、反馈入口、诊断错误、历史数据和设置文案。
- 增加缺失 key 检查，CI 中阻止只改一种语言。
- 截图检查三种语言下的布局溢出、按钮宽度和长文本换行。

### 紧凑胶囊与菜单栏状态

状态：采纳，本轮 P1。

原因：

- 未展开状态不应该给用户很多显示模式选择。
- 常驻信息应直接表达风险和当前用量，不长期显示刷新时间。
- 颜色、状态词和百分比组合，比一整句“能用到几点刷新”更适合常驻。

默认形态：

```text
安全 17%
危险 92%
未知
```

规则：

- 百分比默认使用 5 小时窗口已用比例。
- 如果缺少短窗口或状态未知，不硬凑百分比。
- 菜单栏和未展开胶囊使用同一套显示规则。
- 不提供多种用户可选紧凑显示模式，避免设置负担。

### 展开面板信息增强

状态：采纳，本轮 P1。

原因：

- 展开面板应该解释“为什么是这个状态”，但不能变成大 dashboard。
- 当前“当前速度”下面的进度条没有明确语义，容易让用户误解为 0-100% 进度。

要求：

- 时间进度、额度已用、刷新余量保留进度条。
- 当前速度只显示倍率，例如 `0.22x`，不显示进度条。
- 周预测以低注意力区域展示，不能抢 5 小时主判断。

### 周预测

状态：采纳，本轮 P1。

原因：

- 用户除了 5 小时窗口，也需要低频查看本周压力。
- 周预测不应进入默认胶囊主视觉，只在展开面板中辅助展示。

公式要求：

- 分子必须使用 `weeklyWindow.usedPercent`。
- 分母必须使用周窗口自身的时间进度：`now - (weeklyWindow.resetsAt - weeklyWindow.windowMinutes)`。
- 剩余时间必须使用周窗口自身的 `weeklyWindow.resetsAt - now`。
- 不能复用 5 小时窗口的 `usedPercent`、`elapsedPercent`、`resetsAt` 或 `windowMinutes`。
- 缺少 weekly window 时不显示周预测。
- weekly 已用 100% 时显示危险，不显示安全或未知。

## 本轮评估后再决定

### 刷新按钮

状态：已评估，当前版本不在主胶囊放刷新按钮。

问题：

- 当前主胶囊右侧刷新图标看起来像按钮，但无法独立点击。
- 如果可以做到可靠实时更新，默认不需要用户主动刷新。
- 如果不能做到实时，仍然需要“重新整理 / Retry”入口。

结论：

- 主胶囊不放假按钮。
- 正常状态下不显示刷新图标。
- 刷新中显示 spinner。
- 失败或未知时显示状态图标。
- 菜单栏保留手动刷新。
- 当前不承诺严格实时更新，继续使用启动读取 + 每 60 秒轮询。

原因：

- `codex app-server` 可能出现 rate-limit changed notification，但作为产品默认实时机制还没有足够稳定性证据。
- 长连接方案需要处理 app-server 退出、超时、睡眠唤醒、Codex 同步和重连。
- 事件触发后仍应主动调用 `account/rateLimits/read`，不能只信任 notification。

详细记录见 `docs/research/realtime-rate-limit-events.md`。
