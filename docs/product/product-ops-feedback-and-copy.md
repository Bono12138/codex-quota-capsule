# 产品运营、反馈与文案原则

日期：2026-07-01

## 基本原则

Quota Capsule 的默认体验必须遵循 less is more。

所有新增功能和 UI 变动都要谨慎处理，不能因为竞品有、技术上能做、或者看起来信息更多就加入。默认界面只保留能帮助用户更快判断“现在还能不能继续干活”的内容。

未经 owner 明确确认，不添加新的主界面功能、不改变默认显示形态、不引入新的视觉组件。

## 不采用的方向

### 环形用量图

当前不采用环形用量图作为默认功能。

原因：

- 环形图只能表达一个百分比。
- Quota Capsule 的核心判断不是“剩多少”，而是“时间进度”和“额度已用”谁跑得更快。
- 现有上下两个进度条更符合产品核心：用户一眼能看出自己是用快了还是用慢了。

如果以后要加入环形图，只能放在非默认的分析页或实验页，不能替代主判断。

### 阈值设置

当前不把阈值设置作为默认能力。

原因：

- 固定阈值适合传统监控工具，不适合当前“节奏判断器”的主线。
- 产品应该自动根据时间进度、额度已用、burn rate 和刷新时间判断风险，而不是把判断压力交给用户。

## 当前状态体系

### 用户可见状态

- `读取中`：启动后或刷新中，且还没有成功数据。
- `安全`：当前速度能撑到刷新，并且预计余量较充足。
- `注意`：当前速度能撑到刷新，但预计余量很薄。
- `危险`：当前速度撑不到刷新，或者额度已经用完。
- `未知`：数据缺失、读取失败、短窗口缺失、reset 时间异常，或窗口刚刷新且速度还不稳定。

### 数据来源状态

- `实时读取成功`：最近一次刷新成功。
- `显示上次成功数据`：当前刷新失败，但还有上次成功数据可展示。
- `读取失败`：没有可用成功数据。

### 极端情况必须覆盖

- 5 小时额度已用 100%。
- 周额度已用 100%。
- 只有周额度，没有 5 小时窗口。
- Codex CLI 不存在。
- Codex CLI 存在但 app-server 超时。
- Codex 未登录。
- 数据返回字段缺失。
- reset 时间已过期。
- 刷新失败但有上次成功数据。

额度用完时，如果数据源能正常返回，产品应该显示 `危险`，不能显示 `未知`。如果显示 `未知`，说明是数据没读到，不代表额度真的用完。

## 文案与语气

产品可以在不改变状态判断逻辑的前提下，让状态文案更有变化。

目标：

- 用户每次打开时能感觉到产品在工作。
- 文案可以有一点幽默感，但不能影响理解。
- 幽默文案不能替代关键数字和风险判断。

候选方向：

- 额度很多、使用很慢：`放心用，不用省着。`
- 刚刷新或几乎没用：`粮仓满着，先放开干。`
- 速度偏快但还能撑到：`悠着点儿，地主家也没余粮。`
- 接近危险：`这速度有点猛，先别开大工程。`
- 已经危险：`快见底了，先收一收。`
- 数据读取失败：`我也想看，但现在没读到。`

这些文案属于候选，不默认上线。上线前需要 owner 确认语气、边界和展示位置。

## 语音与趣味反馈

后续可以探索搞笑语音或轻量音效，但必须谨慎。

原则：

- 默认关闭。
- 必须可关闭。
- 不打扰工作。
- 不在公共场合造成尴尬。
- 只在明确的状态变化或用户主动触发时出现。

语音不属于 MVP，进入后续候选功能。

## 数据收集原则

数据收集的目标不是“尽可能少”，而是“只收集对产品改进必要、且能解释清楚的数据”。

原则：

- 提前告知。
- 获取用户同意。
- 开源说明采集项。
- 不上传 prompt、session 正文、auth token、API key、cookie。
- 不上传可反推用户具体工作内容的数据。
- 原始额度数据默认本地保存；如要上传，必须单独说明并征得同意。
- 用户可以关闭数据上报。

数据收集要服务于产品优化、问题定位、满意度理解和后续开源传播，而不是监控用户。

## 可以考虑的运营指标

### 开源传播指标

- GitHub stars。
- forks。
- contributors。
- issues。
- pull requests。
- release 下载量。
- 文档访问量。
- 社交媒体提及量。

### 产品使用指标

- 日活、周活、月活。
- 启动次数。
- 常驻胶囊开启率。
- 展开详情次数。
- 手动刷新次数。

- 数据源成功率。
- D1 / D7 / D30 留存。

- 用户是否更少打开 Codex 官网查额度。

- 用户满意度。
- NPS 推荐意愿。
- 问题上报解决率。

## 问题事件上报

后续需要加入问题事件上报能力，用于收集：

- Bug。
- 改进建议。
- 视觉/交互问题。
- 数据读取失败。
- 安装问题。
- 极端额度状态显示错误。

设计原则：

- 用户主动触发。
- 自动附带最小诊断信息，例如 app 版本、macOS 版本、数据源状态、最近错误类型。
- 不附带 prompt、session、auth、cookie。
- 提交前让用户看到将要发送的内容。

## 更新提醒

后续需要设计版本更新提醒。

候选方式：

- 菜单栏提示。
- 胶囊详情页提示。
- 系统通知。
- 独立更新弹窗。

默认原则：

- 不打断正在工作。
- 重大 bug 修复可以更明显。
- 普通版本更新放在菜单栏或详情页里。
- 用户自己选择是否更新。

当前未决定具体形式，后续设计前需要 owner 确认。

## 近期已发现问题

- 菜单栏 `显示/隐藏悬浮胶囊` 点击后没有明显变化。原因初步判断是 SwiftUI 菜单里通过 `NSApp.delegate` 间接查找控制器不稳定，且菜单打开时曾经主动 `show()` 胶囊，容易抵消隐藏状态。修复方向：把 toggle 回调从 App 入口显式传入菜单内容，并移除菜单打开时的自动 show。
- 刷新图标看起来像按钮，但当前不是独立可点击按钮，容易造成“假按钮”感。后续需要 owner 决定：做成真正按钮，或去掉按钮感。
